# Adjusta 要件定義書

## 1. 概要

### 1.1 サービス名

Adjusta

### 1.2 目的

Adjusta は、Google Calendar と連携し、日程調整における候補日程の作成、管理、確定、メール文面作成を支援する Web アプリケーションである。

ユーザーが自身のカレンダー予定を確認しながら候補日程を作成し、調整相手へ提示するためのメール文面を生成できるようにすることで、日程調整にかかる手間を削減する。

### 1.3 背景・課題

日程調整では、以下のような作業が発生する。

- Google Calendar などで空き時間を確認する
- 候補日程を複数洗い出す
- 候補日程をメールやチャットに転記する
- 相手からの返信をもとに予定を確定する
- 確定した予定をカレンダーに登録する
- 不要になった候補予定を削除・整理する

これらの作業は手動操作が多く、転記ミスや管理漏れが発生しやすい。Adjusta では、Google Calendar と連携し、候補日程の登録・管理・確定・メール文面作成を一連の流れとして扱う。

## 2. スコープ

### 2.1 対象範囲

Adjusta の初期開発における対象範囲は以下とする。

- Google OAuth による認証
- Google Calendar 連携
- ユーザーのカレンダー一覧取得
- カレンダー予定の取得
- 日程調整イベントの作成
- 複数の候補日程の登録
- 候補日程の優先順位管理
- 候補日程を Google Calendar に登録
- 候補日程の編集・削除
- 候補日程から確定日程を選択
- 確定日程を Google Calendar に登録または更新
- 不要になった候補予定の整理
- 候補日程を含むメール文面の作成
- イベント一覧・詳細・編集画面の提供

### 2.2 対象外

初期開発では以下を対象外とする。

- Google Calendar 以外の外部カレンダー連携
- 複数ユーザー間でのリアルタイム共同編集
- アプリ内チャット
- メールの自動送信
- 調整相手側の回答フォーム
- 決済機能
- 組織管理機能
- 高度な通知機能
- 海外タイムゾーン対応

ただし、将来的な拡張を妨げない設計とする。

## 3. 想定ユーザー

### 3.1 主なユーザー

- Google Calendar を利用している個人ユーザー
- メールで日程調整を行うユーザー
- 複数の候補日程を提示する機会が多いユーザー

### 3.2 利用シーン

- 面談日程の調整
- 打ち合わせ日程の調整
- 研究室・学校・業務上の予定調整
- 個人間の日程候補提示

## 4. 利用フロー

### 4.1 基本フロー

1. ユーザーが Google アカウントでログインする
2. Google Calendar へのアクセス権限を許可する
3. アプリ上で自身のカレンダー予定を確認する
4. 日程調整イベントを作成する
5. 複数の候補日程を選択する
6. 候補日程同期が有効な場合、候補日程を Google Calendar に仮予定として登録する
7. 候補日程を含むメール文面を作成する
8. ユーザーがメール文面をコピーして相手へ送信する
9. 相手からの返信をもとに確定日程を選択する
10. 確定日程を Google Calendar に本予定として登録する
11. 不要になった候補予定を削除または整理する

### 4.2 候補日程編集フロー

1. ユーザーが作成済みイベントの編集画面を開く
2. 基本情報または候補日程を変更する
3. 候補日程同期が有効な場合、追加された候補日程を Google Calendar に作成する
4. 候補日程同期が有効な場合、変更された候補日程に対応する Google Calendar の予定を更新する
5. 候補日程同期が有効な場合、削除された候補日程に対応する Google Calendar の予定を削除する
6. アプリ側のデータベースを更新する

### 4.3 日程確定フロー

1. ユーザーがイベント詳細画面を開く
2. 候補日程の中から確定日程を選択する
3. 選択された日程を確定済みとして保存する
4. Google Calendar 上で確定予定を登録または更新する
5. 確定されなかった候補予定を削除または不要な予定として整理する

## 5. 機能要件

## 5.1 認証機能

### 5.1.1 Google OAuth ログイン

ユーザーは Google アカウントを使用してログインできる。

#### 要件

- Google OAuth を用いて認証する
- 独自のユーザー ID・パスワード認証は持たない
- 自前の JWT 認証は原則として持たない
- 認証状態はセッションまたは OAuth に基づいて管理する
- 未認証ユーザーが保護されたページへアクセスした場合、ログイン画面へリダイレクトする
- 認証済みユーザーがログイン画面へアクセスした場合、ダッシュボード等へリダイレクトする

### 5.1.2 アクセストークン管理

Google Calendar API を利用するため、アクセストークンおよび必要に応じてリフレッシュトークンを管理する。

#### 要件

- Google Calendar API の呼び出しに必要なトークンを保持する
- アクセストークンの期限切れ時はリフレッシュ処理を行う
- リフレッシュトークンが失効した場合は再認可を促す
- トークン情報は安全に保存する

## 5.2 カレンダー連携機能

### 5.2.1 カレンダー一覧取得

ユーザーが利用可能な Google Calendar の一覧を取得する。

#### 要件

- ユーザーが所有または参照可能なカレンダーを取得する
- メインカレンダーとその他のカレンダーを区別できる
- 祝日カレンダーなど、複数ユーザーで利用される共有カレンダーも扱えるようにする
- アプリ側ではユーザーとカレンダーを多対多で管理する

### 5.2.2 予定取得

ユーザーのカレンダー予定を取得し、日程選択時に参照できるようにする。

#### 要件

- 指定期間内の予定を取得する
- 複数カレンダーの予定を表示できる
- 候補日程を選ぶ際に既存予定と重複しないか確認できる
- 日本国内での利用を前提とし、初期段階では海外タイムゾーン対応は行わない

### 5.2.3 候補予定の Google Calendar 登録

候補日程同期が有効な場合、ユーザーが選択した候補日程を Google Calendar に仮予定として登録する。

#### 要件

- 候補日程同期が有効な場合、候補日程ごとに Google Calendar の予定を作成する
- 作成した Google Calendar イベント ID をアプリ側に保存する
- 候補予定であることが分かるタイトルまたは説明を付与する
- 候補予定は Adjusta が管理する専用カレンダーに登録する
- 候補予定の登録先カレンダーはユーザーに選択させない

### 5.2.4 確定予定の Google Calendar 登録

確定した日程を Google Calendar に本予定として登録する。

#### 要件

- 確定日程を Google Calendar に登録する
- 候補予定として作成済みの Google Calendar イベントがある場合、それを更新して確定予定として扱えるようにする
- 確定対象以外の候補予定は削除または整理する
- 確定後に再調整する場合にも対応できる設計とする

## 5.3 イベント管理機能

### 5.3.1 イベント作成

ユーザーは日程調整用のイベントを作成できる。

#### 入力項目

- タイトル
- 説明
- 場所
- 対象カレンダー
- 候補日程

#### 要件

- タイトルは必須とする
- 説明・場所は任意とする
- 候補日程は複数登録できる
- 候補日程には開始日時・終了日時を持たせる
- 候補日程には優先順位を持たせる
- 候補日程同期が有効な場合、登録後の同期処理で Google Calendar に候補予定を作成する

### 5.3.2 イベント一覧表示

ユーザーは作成済みのイベント一覧を確認できる。

#### 要件

- イベントのタイトル、状態、更新日時などを表示する
- 候補日程は一覧では必要に応じて省略する
- 詳細画面への導線を設ける
- 視認性の高いカード形式またはリスト形式で表示する

### 5.3.3 イベント詳細表示

ユーザーはイベントの詳細を確認できる。

#### 要件

- イベント基本情報を表示する
- 候補日程一覧を表示する
- 候補日程の優先順位を表示する
- 確定済みの場合は確定日程を表示する
- 編集画面への導線を設ける
- 日程確定操作を行える

### 5.3.4 イベント編集

ユーザーは作成済みイベントの内容を編集できる。

#### 要件

- タイトル、説明、場所を編集できる
- 候補日程を追加できる
- 候補日程を削除できる
- 候補日程の日時を変更できる
- 候補日程の優先順位を変更できる
- Google Calendar 上の予定とアプリ側データの整合性を保つ

### 5.3.5 イベント削除

ユーザーは作成済みイベントを削除できる。

#### 要件

- アプリ側のイベントを削除する
- 関連する候補予定・確定予定の Google Calendar イベントを削除するか、削除確認を行う
- 削除操作は取り消しが難しいため、確認ダイアログを表示する

## 5.4 候補日程管理機能

### 5.4.1 候補日程登録

ユーザーはカレンダー UI または日時入力 UI から候補日程を追加できる。

#### 要件

- 開始日時と終了日時を指定できる
- 単一の日程範囲を選択できる
- 複数の候補日程を登録できる
- 候補日程は selected date として一時保持し、登録後に proposed date として扱う

### 5.4.2 候補日程の優先順位変更

ユーザーは候補日程の優先順位を変更できる。

#### 要件

- ドラッグアンドドロップで並び替えできる
- 優先順位は数値として保存する
- 内部保存する priority は、数値が大きいほど高優先とする
- ユーザーにとって直感的な順序で表示する
- ユーザーへの表示上は、最も優先度が高い候補を 1 として順位表示できるようにする
- フロントエンド側での優先順位管理を中心とする
- 同一 Event 内で、未削除の候補日程の priority は一意とする
- 並び替え時は疎な priority を用いて効率的に更新し、値が詰まった場合のみ event 単位で再採番する

### 5.4.3 候補日程削除

ユーザーは不要な候補日程を削除できる。

#### 要件

- 削除対象の候補日程を UI 上で明示する
- 登録済み候補日程を削除した場合、対応する Google Calendar イベントも削除する
- 未登録の selected date はフロントエンド上から削除する

## 5.5 日程確定機能

### 5.5.1 確定日程選択

ユーザーは候補日程の中から確定日程を選択できる。

#### 要件

- 候補日程一覧から一つの候補を確定できる
- 確定された候補日程は ProposedDate.status = confirmed とする
- 確定されなかった候補日程は ProposedDate.status = not\_selected とする
- Event.status は confirmed とする
- Event.confirmed\_date\_id に、確定された ProposedDate.id を設定する
- Event.confirmed\_google\_event\_id に、メインカレンダー上に作成した確定予定の Google Calendar Event ID を保存する
- 確定後も再調整できる余地を残す

### 5.5.2 確定後の候補予定整理

確定後、不要になった候補予定を整理する。

#### 要件

- 確定対象以外の Google Calendar 候補予定を削除する
- 確定対象の予定は本予定として更新する
- 既に確定済みの日程を再変更する場合は、不要な Google Calendar イベントを整理する

## 5.6 候補日程共有機能

### 5.6.1 候補日程一覧のコピー

初期開発では、メール作成機能の優先度を下げ、候補日程の一覧をユーザーがコピーできる機能を中心とする。

#### 要件

- イベントに紐づく候補日程を、コピーしやすいテキスト形式で表示する
- 候補日程は日時順または優先順位順で表示できる
- ユーザーが候補日程一覧をコピーして、メール・チャット・その他の連絡手段に貼り付けられる
- 初期段階ではメールの自動生成・自動送信は行わない
- メール文面作成やテンプレート管理は将来的な拡張機能として扱う

## 5.7 アカウント・カレンダー設定機能

### 5.7.1 アカウント情報表示

ユーザーは自身のアカウント情報と Google 連携状態を確認できる。

#### 要件

- プロフィール(アバター・名前・メールアドレス)を表示する
- プロフィールは Google アカウント由来とし、アプリ側では編集しない
- Google 連携状態(正常 / 再認可が必要)を表示する
- 再認可が必要な場合は再認可の導線を表示する
- ログアウト操作を提供する

### 5.7.2 カレンダー設定変更

ユーザーはカレンダーごとの利用設定(UserCalendar)を変更できる。

#### 要件

- ユーザーに紐づくカレンダーの一覧を、役割(role)とともに表示する
- ユーザーはカレンダーごとに予定確認での表示対象(is_visible)を切り替えられる
- ユーザーは確定予定の登録先カレンダー(role = primary)を変更できる
- primary は 1 ユーザーにつき 1 つとし、変更時は旧 primary を参照用(reference)に変更する
- ユーザーは候補予定の Google Calendar 同期(sync_proposed_dates)を切り替えられる
- 初期状態は同期無効とし、ユーザーが有効化するまで Adjusta 専用カレンダーを作成しない
- sync_proposed_dates は Adjusta 専用カレンダー(role = adjusta_candidate)でのみ有効とする
- sync_proposed_dates を無効にしても、Adjusta 専用カレンダー自体は削除しない
- sync_proposed_dates を有効にした時点で、Adjusta 専用カレンダーが存在しなければ作成または再作成し、作成に失敗した場合は有効化しない
- Adjusta 専用カレンダーはアプリが自動管理し、ユーザーによる名称変更・削除は対象外とする

### 5.7.3 退会(将来拡張)

退会(アカウント削除)は初期リリースでは提供せず、将来拡張として扱う。

#### 要件

- 初期リリースでは退会機能・UI を提供しない
- 実施する場合は、削除対象データの範囲、Google Calendar 側の予定・専用カレンダーの扱い、Soft Delete との関係を先に要件定義する

## 6. 非機能要件

## 6.1 セキュリティ

- Google OAuth を利用し、独自パスワードは保持しない
- アクセストークン・リフレッシュトークンは安全に保存する
- ユーザーは自身のデータのみ参照・操作できる
- API では認可チェックを行う
- 環境変数でシークレットを管理する
- 不要なスコープを要求しない

## 6.2 可用性

- Google Calendar API の一時的な失敗に備え、エラーハンドリングを行う
- Google Calendar との同期に失敗した場合、ユーザーに分かる形で通知する
- DB 更新と Google Calendar 更新の整合性に注意する

## 6.3 保守性

- フロントエンドとバックエンドを分離する
- 認証、カレンダー、イベント、メール作成などの機能単位でディレクトリを分ける
- ビジネスロジックを handler から分離する
- Google Calendar 操作は専用の manager または service に集約する
- DB 操作は repository に集約する
- トランザクション境界を明確にする

## 6.4 拡張性

- 将来的に調整相手用回答フォームを追加できる設計とする
- 将来的にメール送信機能を追加できる設計とする
- 将来的に候補予定専用カレンダーを利用できる設計とする
- 将来的に通知機能を追加できる設計とする

## 6.5 ユーザビリティ

- カレンダーから直感的に候補日程を選択できる
- 候補日程の追加・削除・並び替えを分かりやすくする
- 確定操作の位置を分かりやすくする
- 一覧画面では情報量を絞り、詳細画面で候補日程を確認できるようにする
- エラー発生時にはユーザーが次に取るべき行動を示す

## 7. データ要件

## 7.1 主なエンティティ

### 7.1.1 User

アプリを利用するユーザーを表す。

主な属性:

- ID
- 名前
- メールアドレス
- プロフィール画像URL
- 作成日時
- 更新日時

### 7.1.2 Account

Google アカウントとの連携情報および Google Calendar API 利用に必要なトークン情報を表す。

主な属性:

- ID
- User ID
- Google User ID
- アクセストークン
- リフレッシュトークン
- トークン有効期限
- スコープ
- 作成日時
- 更新日時

### 補足

- 初期実装では 1ユーザーにつき 1 Account を持つ
- Account は Google 前提の論理設計とし、provider は持たない

### 7.1.3 Session

認証状態を管理するログインセッションを表す。

主な属性:

- ID
- User ID
- セッショントークン
- セッション有効期限
- 作成日時
- 更新日時

### 7.1.4 Calendar

Google Calendar のカレンダー情報を表す。

主な属性:

- ID
- Google Calendar ID
- カレンダー名
- 説明
- タイムゾーン
- 作成日時
- 更新日時

### 7.1.5 UserCalendar

ユーザーとカレンダーの関連を表す中間エンティティ。

祝日カレンダーなど、複数ユーザーで同一カレンダーを参照するケースがあるため、User と Calendar は多対多で管理する。

UserCalendar は、単なる中間テーブルではなく、ユーザーにとってそのカレンダーがどのような役割を持つかを管理する。

主な属性:

- User ID
- Calendar ID
- ロール
- 表示設定
- sync_proposed_dates
- 作成日時
- 更新日時

### CalendarRole

UserCalendar のロールは、ユーザーにとってのカレンダー用途を表す。

| role               | 意味                        |
| ------------------ | ------------------------- |
| primary            | 確定予定を登録するメインカレンダー         |
| adjusta\_candidate | 候補予定を表示する Adjusta 専用カレンダー |
| reference          | 予定確認用・参照用カレンダー            |

### 制約

- User と Calendar の組み合わせは一意とする
- 1ユーザーにつき、adjusta\_candidate ロールの UserCalendar は原則1つとする
- 1ユーザーにつき、primary ロールの UserCalendar は原則1つとする
- Adjusta 専用カレンダーは自動作成し、UserCalendar.role = adjusta\_candidate として管理する
- カレンダー用途は role によって表現し、候補予定の Google Calendar 同期有無は UserCalendar.sync_proposed_dates によって表現する
- UserCalendar.sync_proposed_dates は、role = adjusta\_candidate の行でのみ意味を持つ

PostgreSQL では、以下のような制約を検討する。

```sql
CREATE UNIQUE INDEX uq_user_calendar
ON user_calendars (user_id, calendar_id);

CREATE UNIQUE INDEX uq_user_adjusta_candidate_calendar
ON user_calendars (user_id)
WHERE role = 'adjusta_candidate';

CREATE UNIQUE INDEX uq_user_primary_calendar
ON user_calendars (user_id)
WHERE role = 'primary';
```

### 7.1.6 Event

日程調整イベントを表す。

主な属性:

- ID
- User ID
- Primary Calendar ID
- タイトル
- 説明

## 8. Enum 定義

## 8.1 UserCalendar.role

ユーザーにとって、そのカレンダーがどのような用途を持つかを表す。

| 値                  | 意味                        |
| ------------------ | ------------------------- |
| primary            | 確定予定を登録するメインカレンダー         |
| adjusta\_candidate | 候補予定を表示する Adjusta 専用カレンダー |
| reference          | 予定確認用・参照用カレンダー            |

### 補足

- primary は、確定予定の登録先として扱う
- adjusta\_candidate は、Adjusta が自動作成する候補予定用カレンダーを表す
- reference は、予定の確認・空き時間確認に利用する参照用カレンダーを表す

## 8.2 Event.status

日程調整イベント全体の状態を表す。

| 値         | 意味                       |
| --------- | ------------------------ |
| draft     | 作成途中。まだ候補日程を正式に提示していない状態 |
| active    | 候補日程を提示中・調整中の状態          |
| confirmed | 日程が確定済みの状態               |
| cancelled | 日程調整自体を中止した状態            |

### 状態遷移の例

```text
draft -> active -> confirmed
active -> cancelled
draft -> cancelled
confirmed -> active
```

confirmed から active への遷移は、確定後に再調整する場合に利用する。

## 8.3 ProposedDate.status

候補日程1つ1つの状態を表す。

| 値             | 意味               |
| ------------- | ---------------- |
| active        | 現在有効な候補日程        |
| confirmed     | 確定された候補日程        |
| not\_selected | 確定時に選択されなかった候補日程 |
| cancelled     | ユーザーが取り下げた候補日程   |

### 補足

- ProposedDate.status = confirmed は、Event.confirmed\_date\_id が指す候補日程と一致する
- 日程確定時、確定された候補日程は confirmed とする
- 日程確定時、確定されなかった候補日程は not\_selected とする
- is\_finalized は持たず、status = confirmed で確定済み候補を表す

## 8.4 sync\_status

Google Calendar との同期状態を表す。

Event と ProposedDate で共通の enum として扱う。

| 値             | 意味                             |
| ------------- | ------------------------------ |
| not\_synced   | まだ Google Calendar に同期されていない状態 |
| pending\_sync | 同期が必要な状態                       |
| synced        | Google Calendar と同期済みの状態       |
| sync\_failed  | 同期処理に失敗した状態                    |

### 補足

- MVP では上記4状態を基本とする
- Google Calendar 側で削除・変更された場合も、Adjusta 側を正として復元するため、基本的には pending\_sync または sync\_failed として扱う
- 将来的に外部変更の検知状態を明示したい場合、external\_missing / external\_modified の追加を検討する

## 10. 技術的前提

要件定義書では、具体的なライブラリ名やフレームワーク名は原則として扱わない。 使用技術の詳細は、基本設計書・アプリケーション設計書・技術選定書で定義する。

### 10.1 基本方針

- Web アプリケーションとして提供する
- フロントエンドとバックエンドを分離した構成とする
- Google OAuth による認証を利用する
- Google Calendar API と連携する
- データは RDB に保存する
- バックエンドは DDD を意識した層構成とする
- DB アクセスの具体技術は後続の設計書で定義する
- フロントエンドの状態管理・フォーム送信方式は後続の設計書で定義する

### 10.2 バックエンドアーキテクチャ方針

バックエンドは DDD を意識し、主に以下の層に分けて実装する。

#### interface 層

外部からの入力を受け取り、usecase 層へ処理を委譲する。

主な責務:

- HTTP request / response の変換
- 入力バリデーション
- 認証済みユーザー情報の取得
- HTTP ステータスコードへの変換

#### usecase 層

アプリケーション固有の処理手順を表現する。

主な責務:

- イベント作成
- 候補日程登録
- 候補日程更新
- 日程確定
- Google Calendar 同期の orchestration
- トランザクション境界の管理

#### domain 層

Adjusta の業務ルールを表現する。

主な責務:

- Event / ProposedDate / Calendar / UserCalendar などのドメインモデル
- repository interface
- 候補日程の妥当性判定
- 日程確定ルール
- 優先順位ルール
- Adjusta 側を正とする同期方針の表現

#### infrastructure 層

外部技術への依存を扱う。

主な責務:

- DB アクセス
- repository 実装
- Google Calendar API クライアント
- OAuth トークン保存・更新
- 外部 API エラーの変換

### 10.3 依存方向

依存方向は以下を基本とする。

```text
interface → usecase → domain
                 ↓
          infrastructure
```

ただし、domain 層は infrastructure 層に依存しない。 repository interface は domain 側に定義し、infrastructure 層で実装する。

## 12. Google Calendar 同期方針

## 12.1 基本方針

Adjusta 側で作成したイベントおよび候補日程については、Adjusta のデータベースを正とし、Google Calendar は表示・参照先として同期する。

Google Calendar 側で Adjusta 管理下の予定が削除または変更されていた場合でも、原則として Adjusta 側の状態を優先し、次回同期時に Google Calendar 側へ再作成または再更新する。

### 方針

- Adjusta 側の Event / ProposedDate を正とする
- Google Calendar は同期先として扱う
- Adjusta 管理下の Google Calendar イベント ID を DB に保存する
- Google Calendar 側で削除された候補予定は、必要に応じて再作成する
- Google Calendar 側で変更された候補予定は、Adjusta 側の内容で上書きする
- 同期対象は Adjusta が作成した予定に限定する
- Google Calendar 側で作成された通常予定は Adjusta 側で管理しない

## 12.2 候補予定と確定予定のカレンダー配置

候補予定と確定予定は、Google Calendar 上で表示先を分ける。

### Adjusta 専用カレンダー

- 候補日程同期を有効にした時点で、候補予定表示用の専用カレンダーが存在しなければ自動作成する
- 専用カレンダーは候補予定の表示先として利用する
- 既に Adjusta 専用カレンダーが存在する場合は再利用する
- 専用カレンダーの Google Calendar ID は Calendar に保存する
- ユーザーにとっての用途は UserCalendar.role = adjusta\_candidate として管理する
- 候補予定の Google Calendar 同期有無は UserCalendar.sync\_proposed\_dates で管理する
- UserCalendar.sync\_proposed\_dates を true から false に変更しても、Adjusta 専用カレンダー自体は即時削除しない
- UserCalendar.sync\_proposed\_dates を false から true に変更したとき、Adjusta 専用カレンダーが存在しなければその時点で作成または再作成する
- 1ユーザーにつき Adjusta 専用カレンダーは原則1つとする
- ユーザーが Google Calendar 側で専用カレンダーを削除した場合は、必要に応じて再作成する

### 候補予定

- 候補予定は、原則として Adjusta 専用カレンダーに作成する
- 候補予定を Google Calendar に表示するかどうかは、ユーザー設定で選択できるようにする
- UserCalendar.sync\_proposed\_dates = false の場合、以後は Google Calendar に候補予定を新規作成・更新・再作成せず、Adjusta 側の DB のみで管理する
- UserCalendar.sync\_proposed\_dates = true の場合、Adjusta 専用カレンダーへ候補予定を同期する
- UserCalendar.sync\_proposed\_dates の ON/OFF 切り替えだけを理由に、既存の候補予定を Google Calendar から即時削除しない
- 既存の候補予定を Google Calendar から削除する場合は、設定変更ではなく明示的なユーザー操作で行う

### 確定予定

- 確定日程は、ユーザーのメインカレンダーに本予定として作成する
- 候補予定として作成済みの Google Calendar イベントをそのまま本予定にするのではなく、メインカレンダーに確定予定として登録する
- 確定後、候補予定は Adjusta 専用カレンダー上でステータス変更して残す
- 確定対象以外の候補予定も、削除ではなく候補終了・非確定などの状態として残す

## 12.3 同期タイミング

初期実装では、同期タイミングをイベント詳細画面にアクセスしたときとする。

### 要件

- イベント詳細画面を開いた際に、対象 Event / ProposedDate の同期状態を確認する
- UserCalendar.sync\_proposed\_dates = true の場合、候補日程ごとに Adjusta 専用カレンダーへの create / update / recreate を試みる
- UserCalendar.sync\_proposed\_dates = false の場合、イベント詳細画面アクセス時も Google Calendar への候補予定同期は行わない
- Google Calendar 側で候補予定が削除されていた場合、Adjusta 側の内容をもとに再作成する
- Google Calendar 側で候補予定が変更されていた場合、Adjusta 側の内容で上書きする
- Adjusta 専用カレンダーが削除されていた場合、必要に応じて再作成する
- 同期処理に失敗した場合でも、イベント詳細画面自体は可能な範囲で表示する
- 同期失敗時にはユーザーに同期状態を表示する
- 同期成功時は Event / ProposedDate の sync_status, last_synced_at, last_sync_error を更新する
- 同期失敗時は該当 Event / ProposedDate を sync_failed とし、last_sync_error を更新する

### 暫定方針

- 初期実装ではイベント詳細アクセス時の同期を基本とする
- 将来的には手動同期ボタン、定期同期、バックグラウンドジョブによる同期を検討する

## 12.4 操作別同期方針

| 操作          | Google Calendar 側の処理                                             | DB 側の処理                                                                                     |
| ----------- | ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| 専用カレンダー初期化  | Adjusta 専用カレンダーを作成または再利用                                         | Calendar / UserCalendar に保存                                                                 |
| 候補日程追加      | 設定が有効な場合、Adjusta 専用カレンダーにイベント作成                                  | ProposedDate 作成                                                                             |
| 候補日程更新      | 設定が有効な場合、対応イベントを Adjusta 側の内容で更新                                 | ProposedDate 更新                                                                             |
| 候補日程削除      | 対応イベントを削除またはステータス変更                                              | ProposedDate を削除または無効化                                                                      |
| 候補予定の外部削除検知 | Adjusta 側の内容で再作成                                                 | Google Calendar Event ID を更新                                                                |
| 候補予定の外部変更検知 | Adjusta 側の内容で上書き                                                 | 必要に応じて同期状態を更新                                                                               |
| 日程確定        | primary\_calendar\_id が指すメインカレンダーに確定予定を作成し、専用カレンダー上の候補予定はステータス変更 | Event.status / ProposedDate.status / confirmed\_date\_id / confirmed\_google\_event\_id を更新 |
| イベント削除      | 関連する候補予定・確定予定を削除またはステータス変更                                       | Event 削除または無効化                                                                              |

## 12.5 同期状態の管理

Event と ProposedDate には、Google Calendar との同期状態を管理するための情報を持たせる。

### Event に持たせる同期情報

- 同期状態
- 最終同期日時
- 確定予定の Google Calendar Event ID
- 最後の同期エラー内容

### ProposedDate に持たせる同期情報

- 同期状態
- Google Calendar Event ID
- 最終同期日時
- 最後の同期エラー内容

### 同期状態の例

| 状態                 | 意味                        |
| ------------------ | ------------------------- |
| not\_synced        | 未同期                       |
| synced             | 同期済み                      |
| pending\_sync      | 同期待ち                      |
| sync\_failed       | 同期失敗                      |

### 方針

- 初期実装では同期状態を複雑にしすぎない
- 最低限、同期済み・同期待ち・同期失敗を判別できるようにする
- external\_missing / external\_modified は将来的な拡張候補として扱う
- Google Calendar 側の削除・変更は、Adjusta 側の状態で復元するための検知情報として扱う
- Google Calendar 側の変更内容を Adjusta 側へ取り込まない

## 12.6 整合性に関する注意点

Google Calendar API と DB 更新は完全な分散トランザクションにできないため、以下を考慮する。

- Adjusta 側の DB を正とする
- Google Calendar 同期に失敗した場合でも、Adjusta 側の状態は保持する
- 同期失敗時には再同期できるようにする
- Google Calendar イベント ID が無効になった場合は、再作成して ID を更新する
- Google Calendar 側の外部変更を取り込むのではなく、Adjusta 側の状態で復元する
- 同期状態や最終同期日時を保存できる設計を検討する

## 13. 削除・ログ・追跡性の方針

## 13.1 Soft Delete

Adjusta では、サービス全体として Soft Delete を利用する方針とする。

### 要件

- 主要エンティティには deleted\_at を持たせる
- 削除操作では原則として物理削除せず、deleted\_at を設定する
- 通常の取得処理では deleted\_at が設定されているレコードを除外する
- 必要に応じて、管理・復旧・調査用途で削除済みレコードを参照できる設計とする
- Google Calendar 側の予定削除またはステータス変更と、Adjusta 側の Soft Delete の意味を区別する

### 対象候補

- Event
- ProposedDate
- Calendar
- UserCalendar
- EmailTemplate

Account / Session 関連の削除方針は、認証ライブラリやセキュリティ要件に合わせて別途検討する。

## 13.2 操作ログ

初期実装では、操作ログ専用テーブルは持たない。

### 方針

- MVP では AuditLog / OperationLog のような汎用ログテーブルは作成しない
- Event / ProposedDate の status と sync\_status によって、業務状態と同期状態を管理する
- deleted\_at によって削除済み状態を保持する
- created\_at / updated\_at / deleted\_at によって基本的な変更履歴を把握する
- 本格的な操作履歴・監査ログは将来的な拡張とする

## 13.3 同期ログ

初期実装では、Google Calendar 同期ログ専用テーブルも持たない。

### 方針

- MVP では SyncLog テーブルは作成しない
- Event / ProposedDate に sync\_status を持たせる
- Event / ProposedDate に last\_synced\_at を持たせる
- Event / ProposedDate に last\_sync\_error を持たせる
- Google Calendar 同期に失敗した場合は、対象レコードに sync\_failed とエラー内容を保存する
- 同期処理の詳細な履歴が必要になった場合、将来的に SyncLog テーブルを追加する

## 14. エラーハンドリング要件

## 13.1 認証エラー

- 未ログイン時はログイン画面へ誘導する
- トークン失効時は再認可を促す

## 13.2 Google Calendar API エラー

- API 呼び出し失敗時はユーザーに通知する
- 権限不足の場合は再認可を促す
- 一時的な失敗の場合は再試行可能であることを示す

## 13.3 入力エラー

- 必須項目未入力を表示する
- 開始日時が終了日時以降の場合はエラーとする
- 候補日程が空の場合は登録できない
