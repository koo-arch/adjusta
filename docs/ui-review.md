# Adjusta UI レビュー(全画面)

## 1. 概要

### 1.1 目的

本書は、Adjusta の全画面を使用感(UX)の観点でレビューし、課題と改善案を優先順位付きで整理するメモである。改善実装に着手する前の合意形成を目的とする。

### 1.2 前提

- **コンポーネント構成の根本的な変更も許容する**。現構成(例: `DetailCard` の単一カード形式)の維持は前提としない
- shadcn/ui への段階的移行は決定済み(2026-07-09)。本書は既存実装の課題を整理し、再設計・置換の優先順位を決めるためのレビューとして扱う
- `frontend/DESIGN.md` をビジュアル仕様の正、`ui-guidelines.md` をコンポーネント利用・移行規約の正とする。本レビューは改善バックログとして扱う
- 記載の課題はすべてコード読解に基づく(2026-07 時点の実装)

### 1.3 レビュー観点

情報設計 / 導線 / 操作フィードバック / 状態設計(loading・error・empty)/ アクセシビリティ / ダークモード / レスポンシブ / 要件定義書・画面設計書との整合

## 2. 横断課題(全画面に共通する構造的問題)

### 2.1 fetch エラーの無処理【最重要・バグ相当】

ダッシュボードの `NeedsActionDrafts.tsx` / `UpcomingEvents.tsx` は hook から `error` を受け取っておらず、一覧の `EventList.tsx` は `error` を受け取るが未使用。いずれも取得失敗時に `isLoading=false`・`data=undefined` となり、**通信エラーが「イベントはありません」という空状態表示(または無表示)に化ける**。ユーザーはデータ消失と誤認しうる。

**改善案**: 画面・セクション単位で loading / error / empty の3状態を必ず描き分ける共通境界を導入する。現状 `app` 配下に `loading.tsx` / `error.tsx` / `not-found.tsx` は 1 つもなく、Suspense はヘッダーの UserMenu のみ。方向性は次の2案:

- 案A: TanStack Query の状態を受けて loading / error / empty / 内容 を出し分ける共通ラッパー(例: `QueryBoundary`)を `src/components/` に追加し、各セクションで使う
- 案B: ルート単位は App Router の `loading.tsx` / `error.tsx` に寄せ、セクション単位のみ案Aを使う

いずれでもエラー時は「再試行」導線を必ず置く。

### 2.2 ローディング表示の粗さ

データ取得中は各所で `<p>Loading...</p>` の素テキスト(ダッシュボード両セクション、EventList、EventEdit)。スケルトンは `UserMenuSkeleton` のみで、読み込み完了時にレイアウトシフトが起きる。

**改善案**: 主要リスト・カードにスケルトンを用意し、「表示領域の寸法をローディング中も維持する」を規約化する(ui-guidelines へ反映)。

### 2.3 ダークモードが中途半端【決定済み: light 固定(案B)】

`ThemeProvider`(system 追従)と `darkMode: "class"` は設定済みだが、`dark:` バリアントの使用は 4 ファイルのみ。さらに**切替 UI(`ThemeButton`)がどの画面にも配置されておらず**(Storybook のみ)、OS がダークのユーザーには「一部だけ暗い崩れた画面」が見える。

**改善案(二択)**:
- 案A: 全面対応 — 共通コンポーネント(Card / Modal / TextField 等)からダーク対応し、ヘッダーに ThemeButton を配置
- 案B: いったん撤去 — `ThemeProvider` を外して light 固定にし、将来まとめて対応

中途半端な現状が最悪であり、どちらかに倒す判断が必要。

### 2.4 空状態の欠落

空状態表示を持つのはダッシュボード(`EmptyStateCard`)と作成フォームの「日程を選択してください」のみ。**イベント一覧は 0 件だと見出しの下が真っ白**で、初回ユーザーに作成導線が示されない。

**改善案**: 「一覧・セクションは必ず空状態(説明+次アクション導線)を持つ」を規約化し、`EmptyStateCard` を共通コンポーネントへ昇格する。

### 2.5 レスポンシブ戦略の二重化

全体は CSS ブレークポイント(`sm:`〜`lg:`)だが、**フォームだけ `react-responsive` の `useMediaQuery`(JS、`EventForm.tsx`)でレイアウト分岐**しており、SSR とのハイドレーション不整合・初期表示のちらつきリスクがある。

**改善案**: CSS ブレークポイントに一本化する(`EventForm` のカラム構成を CSS Grid の `md:` 分岐で再実装)。

### 2.6 アクセシビリティの基礎欠落

- `TextField` / `TextArea` は `<label>` と input が `htmlFor`/`id` で関連付けられておらず、`aria-invalid` / `aria-describedby` もない
- ログアウトが `href` なしの `<a onClick>`(`UserButton.tsx`)— button であるべき
- カードのクリック領域が div の onClick(`Card isButton`)でネイティブ button ではない
- `DraggableList` のセンサーは Mouse + Keyboard のみで **Touch/Pointer センサーがなく、モバイルで優先順位の並び替えができない懸念**(要件 5.4.2 の主要機能)

**改善案**: フォームプリミティブの label 関連付けと aria 属性付与、ログアウトの button 化、DnD への `PointerSensor` 追加 + 上下移動ボタンの代替手段(ui-guidelines 2.5 の推奨を実装)。

### 2.7 小粒な品質課題(まとめて修正候補)

- タイポ: `LoignPage`(login/page.tsx)、`ma-auto`(UserInfo.tsx)、`DateTImePicker.tsx`(ファイル名)
- 遷移先ハードコード: `DetailCard` の EditButton が `/schedule/draft/${eventID}/edit` を直書き(URL 改名時の障害。screen-design 移行課題 9.1 と関連)
- ログイン画面の Google ロゴを `svgrepo.com` から外部ロード(`LoginButton.tsx`)— ローカル資産化すべき

## 3. 画面別レビュー

### 3.1 イベント詳細(`features/events/detail/`)【最優先で再設計】

#### 現状

`DetailCard` 1 枚のカードに、タイトル+編集アイコン / 確定日時 or 候補日程(見出し横に小さな確定アイコンボタン)/ 場所 / 説明 / 右下に削除ボタン、を縦積み。

#### 課題

1. **EventStatus バッジがない** — 調整中か確定済みかは「確定日時」見出しの有無でしか分からない(screen-design 5.6 乖離)
2. **候補日程がプレーンテキスト** — 「第N候補: 日時」のみで、優先順位の変更結果・候補ごとのステータス(confirmed / not_selected)・同期状態が見えない
3. **確定操作の発見性が低い** — この画面の最重要アクションが見出し横の小アイコン(`ConfirmButton` の IconButton)のみ。初見で確定方法が分からない
4. **候補日程一覧のコピーが未実装** — 要件 5.6 で初期リリースの共有主導線とされている機能がこの画面にない
5. **同期状態の表示がない** — sync_status / last_sync_error を表示せず、仮予定登録の失敗に気づけない(要件 12.3 乖離)
6. 削除ボタンがカード右下に浮いており、編集(ヘッダー横)と操作の置き場が分散
7. 説明が空でも「説明」見出しだけ残る
8. ローディングが `Loading...` テキストのみ、ダーク非対応

#### 改善案(構成の再設計)

`DetailCard` 単一カードを廃し、次の3ブロック構成に分割する:

1. **ヘッダー領域**: タイトル + `StatusBadge`(EventStatus)+ アクションバー(編集 / 削除 / その他)。破壊的操作の削除はメニュー内 or 右端に分離
2. **候補日程セクション**(画面の主役): 優先順位・日時・候補ステータス・同期状態を持つリスト表示。上部に **「候補をコピー」ボタン**(テキスト形式で日時順/優先順)と**「日程を確定」の目立つ CTA**(未確定時)。確定済みなら確定日時を大きく表示し、非選択候補は畳んで表示
3. **基本情報セクション**: 場所・説明・登録先カレンダー・同期状態のサマリ。空項目は見出しごと省略 or「未設定」で統一

`ConfirmButton` のモーダル(候補から選択 / 手動入力)は流用可能。遷移先ハードコードは解消する。

### 3.2 ダッシュボード(`features/dashboard/`, `features/calendar/`)

#### 課題

1. fetch エラーの無処理(2.1)— 両セクションとも失敗が空表示に化ける
2. **閲覧用カレンダーが編集可能に見える** — `Calendar.tsx` が `selectable` / `editable` = true のままで、ドラッグでイベントを動かせるが保存されない(ハンドラが空)。「操作できるのに何も起きない」は見た目の嘘
3. スライダー(`BoardSlider`)にページャ・件数表示がなく、隠れているカードに気づきにくい
4. ローディングが素テキストでセクション見出しごと消える(2.2)

#### 改善案

- ダッシュボードのカレンダーは閲覧専用(`selectable/editable` を false)にし、クリックで詳細モーダルのみ残す
- スライダーはページネーションドット or「すべて見る」リンク(→ 一覧)を付ける。件数が少ない機能なら、スライダーをやめて縦リスト+「もっと見る」の方が把握しやすい可能性あり(構成変更候補)

### 3.3 イベント一覧(`features/events/components/list/`)

#### 課題

1. **空状態がない**(2.4)— 0 件で真っ白
2. エラーが空表示と区別できない(2.1)
3. 絞り込み・並べ替え・検索がない — ステータス混在のカードが返却順で並ぶだけ。検索 API(`GET /api/event/draft/search`)は既存
4. 画面内に作成導線がない(ヘッダーの「+」のみ)

#### 改善案

- 空状態(説明+「イベントを作成」CTA)を追加
- 最低限の**ステータス絞り込みタブ**(すべて / 調整中 / 確定済み / 下書き)を追加(`ToggleButton` 流用可)。フル検索は Could のまま後続
- 一覧ヘッダー右に作成ボタンを常設

### 3.4 イベント作成・編集(`features/events/draft/`, `edit/`, `components/form/`)

#### 課題

1. **候補日程の追加がカレンダードラッグ一択** — 日時の直接入力手段がなく(`DateTimePicker` は存在するが未使用)、離れた日付の指定はスクロールが延々必要。`timeGridWeek` 固定でビュー切替もない
2. **編集の保存後フローが不整合** — 作成は詳細へ遷移するが、編集はトーストのみで画面に留まり、詳細へ戻る導線がフォーム内にない
3. ボタン文言が編集でも「登録する」
4. 「候補日程の確定」トグル(編集のみ)の意味・副作用が説明なしで伝わらない
5. `useMediaQuery` の JS 分岐(2.5)
6. 入力途中の離脱警告がない
7. モバイルで優先順位 DnD 不能の懸念(2.6)

#### 改善案

- 「選択日程」カードに**「日時を直接追加」ボタン**(DateTimePicker のモーダル/インライン)を併設し、カレンダー選択と両対応にする。カレンダーは週/月ビュー切替を有効化
- 編集の保存成功時は詳細へ遷移(作成と統一)し、文言を「保存する」に変更。フォーム上部にパンくず or「詳細へ戻る」を設置
- 確定トグルには説明文(「ONにすると先頭の候補が確定日時になります」等)を付けるか、確定操作は詳細画面に一本化してフォームから外す(操作の置き場を「編集=内容変更、詳細=確定」に分離する構成変更案)

### 3.5 ログイン(`features/auth/`)

#### 課題

1. **エラー表示が皆無** — OAuth 失敗・キャンセル時にこの画面で受け取る仕組みがない(要件 8.1「エラー表示」乖離)
2. 押下後のフィードバックがない(disabled/スピナーなし)— 遅延時に無反応に見え、二度押しできる
3. Google ロゴが外部 URL 依存(2.7)

#### 改善案

- コールバック失敗時に `/login?error=...` へ戻し、query param でエラーメッセージを表示する
- クリック後はボタンを disabled + スピナー表示

### 3.6 LP(`(marketing)/`)

#### 課題

- 訴求コンテンツが実質ゼロ(キャッチ+画像+CTA の1画面のみ)。機能紹介・利用イメージがなく、初見ユーザーがサービスを理解できない
- フッターに利用規約・プライバシーポリシー等のリンクがない

#### 改善案

- 「3ステップの使い方(候補を選ぶ→共有する→確定する)」と主要機能 3 点のセクションを追加(スコープは要判断)
- フッターに規約類のリンク枠を用意(文書自体の整備は別課題)

### 3.7 アカウント(`features/auth/`)

#### 課題

- アバター・名前・メールの表示のみで操作が何もない。screen-design 5.8 が定義するカレンダー設定(role / is_visible / sync_proposed_dates)は未実装(既知の移行課題 9.5、バックエンド API も未整備)

#### 改善案

`screen-design.md` 5.8 の詳細設計(2026-07-09)に従う:

- Phase 1(表示系): プロフィール / Google 連携状態 + 再認可導線 / ログアウト(既存 API のみで実装可)
- Phase 2: カレンダー設定セクション(`GET/PATCH /api/user-calendars` の新設とセット)

## 4. 実装順序の方針

このドキュメント群を main に取り込んだ後は、画面・UI の再設計に着手する前に、画面改善の前提となる API 契約を先に整備する。

理由は、イベント一覧・アカウント設定・候補日程同期設定などは API の入出力に強く依存し、UI だけを先行すると仮実装や後戻りが発生しやすいためである。

### API 先行で整備する項目

| 優先 | 項目 | 目的 | 関連ドキュメント |
| --- | --- | --- | --- |
| 1 | カレンダー設定一覧取得 API | `/account` で role / is_visible / sync_proposed_dates を表示する | requirements 5.7 / screen-design 5.8 |
| 2 | カレンダー設定更新 API | 候補日程を Google Calendar に表示するか、確定予定の登録先を変更できるようにする | requirements 5.7 / screen-design 9.5 |
| 3 | イベント一覧 pagination | 件数増加に備え、一覧画面のデータ取得契約を固定する | requirements 9.3 / ui-review 3.3 |
| 4 | イベント検索・ステータス絞り込み API | 一覧 UI の検索・絞り込み導線の前提を作る | requirements 9.3 / ui-review 3.3 |

### 推奨ブランチ順

1. `docs/ui-design-guidelines` を main に merge
2. API 整備ブランチを作成し、backend の API / DTO / usecase / repository と frontend の API client / types / queryKeys まで接続する
3. UI 改善ブランチで `frontend/DESIGN.md`、`docs/ui-guidelines.md`、本レビューの P1 から画面を整える

API 整備後に UI を進めることで、一覧・アカウント・詳細画面の再設計時に仮の型や暫定導線を減らす。

## 6. ui-guidelines.md への反映候補

優先順位の合意後、以下を規約として追記する:

1. **状態設計**: データを表示するセクションは loading(スケルトン)/ error(再試行導線付き)/ empty(次アクション導線付き)の3状態を必ず持つ
2. **主要アクション**: 画面の主目的となる操作(確定・登録など)はアイコンのみにせず、ラベル付きの目立つボタンで置く
3. **操作可能性の一致**: 操作できるように見える UI は実際に操作できること(閲覧用カレンダーは編集無効化)
4. **レスポンシブ**: レイアウト分岐は CSS ブレークポイントに一本化し、JS のメディアクエリは使わない
5. **フォーム**: label と入力の関連付け(htmlFor/id)、エラー時の aria 属性を必須とする
6. **保存後の遷移**: 作成・編集の完了時は対象の詳細へ遷移する(トーストのみで留まらない)
7. **外部リソース**: ロゴ・アイコン類はローカル資産とし、外部 URL 直リンクを禁止する
